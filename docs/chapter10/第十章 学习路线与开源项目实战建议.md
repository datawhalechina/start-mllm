# 第十章 学习路线与开源项目实战建议

> **实战篇**（第 5～10 章）：评测、部署、推理与 Demo、Agent、学习路线收束。本章是实战篇的**路线收束**，不再引入新架构名词。

## 全书主线回扣

**理论篇（1～4 章）**把地图、视觉编码与对齐、生成架构和数据配方说清；**实战篇（5～10 章）**用同一批场景做离线评测（第 5 章）、把链路与观测搭稳（第 6 章）、落成脚本与 Demo（第 7～8 章）、再加感知与工具工作流（第 9 章）。本章把这些收成**可执行的迭代计划**，而不是再堆新概念。自检清单也可对照 [前言](../前言.md) 第九节“顺着正文走完，你大致能独立完成什么”。

**工程速记**：很多小团队一周里会在“补 20 条 JSONL → 跑 `eval_vlm_dataset.py` → 改预处理或换模型 → 再写进第六章日志字段”之间循环；第十章下面的阶段路线，就是把这种循环写进日历，而不是只记在脑子里。

## 一、把知识变成稳定能力

学到这里，最重要的不是再记更多模型名，而是把前九章的内容稳定地落到自己的判断和产出上。你至少要能做到：

- 看见一个新任务时，先判断它更偏感知、OCR、推理还是工程链路问题。
- 拿到一张失败样例时，知道应该先回到哪一章排查。
- 在“数据、评测、部署、Demo、Agent”之间形成一条能反复迭代的小闭环。

如果这三件事开始稳定出现，你就已经从“会读教程”走到了“能独立推进项目”。

## 二、把前九章收成一条四阶段路线

### 第一阶段：把结构讲清楚

目标：

- 能用自己的话讲清视觉编码器、对齐层、LLM 的关系
- 看懂主流 VLM 架构图
- 知道为什么同样是多模态模型，效果会差很多

### 第二阶段：把调用跑通

目标：

- 本地或通过 OpenAI 兼容接口跑通一次图文问答
- 知道 `messages`、图像输入和返回结果的基本结构
- 能排查最常见的环境、路径和协议问题

### 第三阶段：把评测和 Demo 接起来

目标：

- 做一份自己的小型评测集
- 能用同一套请求习惯同时跑评测脚本和 Demo
- 记录典型失败样例，并能复现它们

### 第四阶段：把工作流串起来

目标：

- 把感知、工具、规则和结果汇总连接成一个最小工作流
- 能说清什么时候该直接回答，什么时候该调工具，什么时候该要求重传或人工兜底
- 能根据失败样例回补数据、评测和部署侧问题

## 三、做教程项目时最容易忽略的事

### 1. 保持目录稳定

教程仓库最怕目录频繁重构，最好尽早固定：

- 根 README
- docs 章节
- code 示例
- extra 或 examples 扩展区

### 2. 一章只讲一个主问题

不要让单章既讲理论又讲所有工程细节，容易失焦。

### 3. 每个代码示例都要有清晰边界

读者应该能一眼知道：

- 这个脚本做什么
- 依赖是什么
- 怎么运行
- 适合改造成什么

### 4. 给读者一个明确的阅读和练习入口

读者应该知道：

- 先看哪几章
- 先跑哪个脚本
- 第一次实践应该留下什么产出

## 四、一个可直接执行的 30 天练习框架

如果你想把这份教程真正转成自己的能力，可以按下面的节奏推进：

- 第 1 周：读完第 1～3 章，写一页自己的“多模态系统速记卡”
- 第 2 周：完成第 4～5 章最小练习，至少留下 1 份 JSONL 和 1 份评测结果
- 第 3 周：跑通第 7～8 章脚本与 Demo，把同一张图在脚本和界面里都测一遍
- 第 4 周：完成第 9～10 章最小工作流，总结 3 类最常见失败和你的应对策略

## 五、建议你尽早形成自己的“模型判断框架”

真正的工程能力，不是会背模型名，而是能快速回答这些问题：

1. 这个模型的视觉侧强不强？
2. 它擅长自然图像、文档还是截图？
3. 它的输入格式和部署方式是否适合我的系统？
4. 如果回答不准，问题更可能出在什么地方？

当你能稳定回答这些问题时，你就已经从“调用者”进入“构建者”阶段了。

## 六、三条常见的继续实践路线

### 路线 1：场景验证

围绕一个真实场景反复迭代，例如截图助手、文档抽取或商品图问答。重点是把数据、评测、Demo 和工作流打通。

### 路线 2：模型对比

固定一批问题和图片，对比不同模型在你自己的分布上怎么失败。重点是记录差异，而不是追求一次性得到“最强模型”。

### 路线 3：工作流深化

在单轮图文问答之上加入 OCR、规则、检索、缓存或人工兜底，让系统更接近真实交付形态。

这三条路线没有绝对高低，但最好先选一条主线，避免什么都碰一点、最后没有稳定产出。

## 七、章末练习

**动机**：收官要把九章收成你自己能执行的迭代主线；练完你清楚接下来 30 天优先补场景、评测还是部署，仓库不会无限发散。

### 必做（约 15 分钟）

1. 选择“场景验证、模型对比、工作流深化”三条路线中的一条，说明你为什么适合它。

### 进阶（约 40 分钟）

1. 给自己写一个 30 天学习与实践计划，按周拆成可检查的产出。
2. 列出你下一步最想补的 3 个能力点，并说明它们对应哪一章。

### 挑战（1～2 小时）

1. 选一个真实场景，写出“输入是什么、最容易失败在哪里、你准备用哪几章内容解决”。

## 八、最后收束一下

这份教程的最佳用法，不是读完就结束，而是继续把它演化成：

- 你的知识框架
- 你的最小实验集
- 你的可运行 Demo
- 你的第一个多模态工作流

真正有价值的，不是仓库看起来多完整，而是你能不能用它反复解决问题、积累判断、留下产出。
