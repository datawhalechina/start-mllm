# 前言

## 一、为什么要系统学习多模态大模型

大语言模型解决的是“基于文本进行理解和生成”的问题，而多模态大模型（MLLM, Multimodal Large Language Model）试图进一步处理图片、文档、截图、表格、视频帧、语音等多种输入形式。它的真正价值，不是“多接了一个图片入口”，而是让模型逐步具备接近真实世界任务的感知能力。

现实应用里，用户很少只提供纯文本。更常见的输入是：

- 一张报错截图，外加一句“这是什么意思”
- 一份发票或合同，外加一句“帮我提取关键信息”
- 一张商品图，外加一句“帮我写标题和卖点”
- 一段视频封面、多张图片、语音说明、结构化表格的组合输入

如果你只会文本模型，你能做的事情会被输入形式限制；如果你理解了多模态模型，你就能开始做真正贴近业务的 AI 系统。

### 先有一张总地图

如果你刚开始学多模态，先把这条路线记在脑子里：

`概念地图 → 架构原理 → 数据与训练 → 评测与部署 → 推理实操 → Demo → Agent`

后面遇到新术语时，你能立刻知道它属于哪一段，不容易越读越乱。

## 二、本项目适合谁

这份教程主要面向三类读者：

1. 已经学过 LLM，希望进一步理解 VLM/MLLM 的开发者。
2. 做应用或 Agent，希望把图像、文档、截图能力接入系统的工程师。

## 三、阅读前最好具备这些基础

**最低门槛**：Python 基础语法 + 能打开命令行。**有则更好**：PyTorch/Transformers 经验、对 Transformer/Token/Embedding 的初步认知。详细的环境与前置要求请见 [Chapter 0 环境准备](./chapter0-环境准备.md)。

不熟悉 PyTorch 或 Transformers 也没关系。先按”概念优先、代码其次”读，[LLM 基础速通](./llm-基础速通.md) 会帮你快速补齐核心概念。没有 GPU 也能学——第七章提供”纯 API 方案”，有网就能跑通全部实战代码。

### 首读术语速查

- `MLLM / VLM`：能够同时处理视觉与语言输入的模型（VLM 通常特指图文方向）。
- `视觉编码器`：把像素变成向量特征的模块，相当于“眼睛”。
- `Patch Token`：图像切块后形成的视觉 token，粒度影响细节保留能力。
- `Connector / Projector`：把视觉特征映射到 LLM 可消费表示的桥接层。
- `SFT`：指令微调，让模型更像“会执行任务的助手”而不仅是会理解图文关系。
- `Grounding`：让模型的文字描述能定位到图像中的具体区域（如坐标框、边界框），而不只是笼统地描述整张图。

第一次看时先记“这个词是干什么的”就够了，细节可以边做边补。

### 建议尽早做一次最小体验

如果你是“先跑起来再理解”的学习风格，读完第一章就可以马上做这件事：

1. 准备任意一张截图或商品图。
2. 按第七章示例配置 `OPENAI_BASE_URL`、`OPENAI_API_KEY`、`MODEL_ID`。
3. 运行最小推理脚本并提 2 个问题：一个描述类、一个细节类。
4. 记录两类问题的回答差异（是否遗漏细节、是否依赖图像证据）。

这一步不追求效果最好，核心是先建立手感：同一张图，不同问题，模型到底会怎么变。

### 学习者视角：第一周可以怎么迭代（不停下来版）

如果你把自己当成“边学边提交”的读者，建议按最小闭环推进，每一小步都有可检查产出：

1. **数据格式手感**：在 `docs/chapter4/code` 运行 `create_placeholder_images.py`，再跑 `prepare_dataset_manifest.py`，确认 `image_exists` 为真——避免 JSONL 自嗨、路径全断。
2. **评测脚本手感**：在 `docs/chapter5/code` 同样生成占位图，`pip install` 后跑 `eval_vlm_dataset.py`（需可用的 OpenAI 兼容端点）；先看 `status` 与 `error` 字段，再谈准确率。
3. **推理手感**：第七章任选本地 `Transformers` 或 OpenAI 客户端，用上一小步同一张占位图或你自己的截图，问“描述类 + 细节类”各一题，把差异记进你的评测 JSONL 备注里。
4. **Demo 手感**：第八章起 Gradio，把请求体字段习惯和第五章对齐，避免“脚本里对、界面里错”。

卡住时优先对照第二章 Debug 顺序（输入 → 预处理 → 协议 → 模型 → 提示），比换模型名更有效。

## 五、学习时要避免的几个误区

### 误区 1：把“能收图片”当成“理解图像”

很多多模态模型的能力边界并不来自“有没有图片输入”，而来自：

- 视觉编码器是否足够强
- 对齐是否足够好
- 数据分布里是否包含你关心的任务
- 推理链路里是否保留了必要的细粒度信息

### 误区 2：只看排行榜，不看任务结构

一个模型在综合榜单上更高分，不代表它一定适合你的业务。文档理解、OCR 密集场景、中文截图、图表推理、多图对话，都可能需要不同的模型偏好。

### 误区 3：只会调 API，不理解系统结构

如果你只停留在“上传图片然后拿答案”，很难真的判断问题出在：

- Prompt
- 数据
- 模型架构
- 推理配置
- 部署服务

理解结构，才有能力真正定位问题。

## 七、预热练习

### 必做（5-10 分钟）

1. 写下你最常见的 1 个“图像 + 文本”任务场景（如截图报错、票据抽取、商品图文案）。
2. 判断这个场景中最关键的输入证据是什么（小字、布局、图标、上下文描述等）。

### 进阶（20-30 分钟）

1. 画出该场景的最小链路：输入 → 编码 → 对齐 → 生成 → 输出。
2. 写下你最担心的 3 个失败点，并说明你打算如何观测它们。

### 挑战（1-2 小时）

1. 完成一次“10 分钟最小体验”，并记录两类问题的回答差异（描述类 vs 细节类）。
2. 形成一页学习目标卡：你希望在第 4 周前做出什么 Demo、验证什么能力边界。
