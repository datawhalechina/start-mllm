# 第二章 视觉编码器与跨模态对齐

> **理论篇**（第 1～4 章）：地图、视觉编码与对齐、生成架构、数据与微调。

承接 [第一章](../chapter1/第一章%20多模态大模型概览.md) 的模块地图：本章把“眼睛”和**跨模态对齐**拆细；你在这里理解的 patch、分辨率与 Debug 顺序，会直接体现在 [第五章](../chapter5/第五章%20评测体系与工程选型.md) 离线评测里（小字、糊图）和 [第六章](../chapter6/第六章%20推理部署与%20Serving.md) 预处理里（缩放、切块）。

## 一、为什么图像不能直接输入 LLM

语言模型习惯处理的是离散 token 序列，例如：

```text
["这", "是", "一", "张", "猫", "的", "图", "片"]
```

而图像本质上是一个像素张量，例如 `H x W x C`。它既不是离散 token，也没有天然的线性顺序。要想让 LLM 读懂图像，第一步必须先把图像编码成一组可计算的视觉 token。

这就是视觉编码器的职责。

## 二、视觉编码器在系统里的角色

视觉编码器的核心任务是：

1. 从像素中提取语义特征。
2. 保留必要的空间关系和局部细节。
3. 输出一个后续模块能使用的特征序列。

对多模态系统来说，它相当于“眼睛”。如果这双眼睛看得不清，后面的语言模型再强，也只是在对模糊信号做生成。

本章属于**理论篇**，但你在**实战篇**第五～七章会反复用到同一套直觉：同一失败样例，先问“是不是分辨率/patch 让证据没进来”，再问“是不是 prompt 或协议”——顺序错了会白调很久。

## 三、从 CNN 到 ViT

早期视觉模型更多依赖 CNN，优势是局部归纳偏置强，适合做分类、检测、分割。但在大模型时代，Transformer 风格的视觉编码器更容易与 LLM 体系接轨。

Vision Transformer（ViT）的关键思路很简单：

1. 把图像切成很多固定大小的 patch。
2. 把每个 patch 展平成向量。
3. 像处理文本 token 一样处理这些 patch token。

例如，一张 `448 x 448` 的图片，如果 patch size 是 `14 x 14`，那大致会得到 `32 x 32 = 1024` 个 patch token。分辨率越高，token 越多，感知越细，但计算成本也越高。

### 从像素到 token：按工程视角走一遍

可以把这个过程拆成四步：

1. `Resize / Pad`：把输入统一到模型接受的尺寸范围。
2. `Patchify`：按固定窗口切块（如 `14 x 14`）。
3. `Linear Projection`：每个 patch 映射到同维度向量。
4. `Transformer Encoder`：建模 patch 间关系，得到视觉 token 序列。

我自己的经验是，最容易出问题的是前两步：尺寸策略和切块粒度。这一步直接决定了 [第五章](../chapter5/第五章%20评测体系与工程选型.md) 离线评测里“小字遗漏”问题的严重程度。

## 四、Patch Token 为什么重要

多模态模型经常在以下能力上拉不开差距：

- 能不能看清小字
- 能不能看懂复杂版面
- 能不能定位特定区域（即 Grounding，在图中框出目标的位置）

根本原因之一，就是视觉 token 的粒度。粒度越粗，细节越容易丢；粒度越细，算力和显存越高。

因此在工程里你会经常看到这些折中：

- 限制输入分辨率
- 动态裁剪图像
- 只保留部分 patch token
- 对高分辨率图像先做 tile 切分再编码

> **说明：Patch 和 Tile 的区别**
> - **Patch** 是针对单张图片/网格的底层切块处理，比如把 224x224 图像切成 16x16 的像素小块（Patch），每个 Patch 映射成一个 Token。
> - **Tile** 是针对超高分辨率图片（如 4K）的宏观切块策略。先把大图切成几个子图（Tile），再对每个 Tile 分别进行 Patch 切块编码。这样能保留局部高清细节。

### 分辨率与成本速查

| 输入分辨率 | token 数趋势 | 细节保留 | 计算/时延 |
| --- | --- | --- | --- |
| 低 | 少 | 容易丢失小字与边角信息 | 低 |
| 中 | 中 | 多数场景可用 | 中 |
| 高 | 多 | 更有利于 OCR/细节定位（Grounding） | 高 |

实操时通常先用 448x448 中等分辨率跑通，再按任务慢慢往上加，不容易一开始就卡在算力上。

## 五、跨模态对齐在解决什么问题

即使视觉编码器已经把图片变成了向量，这些向量也不一定和语言语义对得上。对齐的目标是让：

- 图像里的“猫”靠近文本里的“猫”
- 图像里的“红色按钮”靠近文本里的“红色按钮”
- 图像整体语义能够被语言模型利用

换句话说，对齐是在回答：**视觉特征怎样才能变成语言模型有意义的上下文。**

## 六、CLIP 是理解对齐最好的起点

CLIP 的核心训练方式是图文对比学习（对比学习是一种训练方式——把匹配的图文对拉近、不匹配的推远，让模型学会区分"像/不像"）。它有一个图像编码器和一个文本编码器（这里的"编码器"在论文中常被称为 tower，指模型的一个独立处理分支），训练目标是：

- 拉近正确图文对
- 拉远不匹配图文对

你可以把它想成“让图片和文字在同一个语义坐标系里站稳位置”。

### 一个直观理解

如果模型看到一张“黄色小狗在草地奔跑”的图片，那么它应该让这张图片的向量靠近：

- “一只黄色的小狗”
- “草地上的狗”
- “奔跑的小狗”

同时远离：

- “夜晚的城市街景”
- “一架飞机停在机场”

这种训练方式对检索、分类迁移、跨模态匹配非常有效，也为后来的生成式多模态模型打下了基础。

### 最小可运行代码：用 CLIP 算图文相似度

你不需要训练任何模型，下面不到 30 行代码就能体验“对齐”的效果：

```python
from PIL import Image
import torch
from transformers import CLIPProcessor, CLIPModel

# 1. 加载模型（首次会自动下载，约 600MB）
model = CLIPModel.from_pretrained("openai/clip-vit-base-patch32")
processor = CLIPProcessor.from_pretrained("openai/clip-vit-base-patch32")

# 2. 准备图片和候选文本
image = Image.open("sample_receipt.png")  # 可先运行 docs/chapter4/code/create_placeholder_images.py 生成
texts = ["a receipt", "an invoice", "a screenshot of a website", "a document with text", "a product photo"]

# 3. 编码
inputs = processor(text=texts, images=image, return_tensors="pt", padding=True)
outputs = model(**inputs)

# 4. 计算图文相似度（余弦相似度后 softmax）
probs = outputs.logits_per_image.softmax(dim=1)

# 5. 输出
for text, prob in zip(texts, probs[0]):
    print(f"{text}: {prob:.4f}")
# 预期输出类似：
# a receipt: 0.52
# an invoice: 0.24
# a screenshot of a website: 0.10
# a document with text: 0.09
# a product photo: 0.05
```

**这个代码在说明什么**：
- CLIP 把“图片”和“文字”都变成了同一个语义空间里的向量
- “票据 / 发票”这类文本向量，和票据图片的向量距离会更近
- 这个能力是后续所有生成式多模态模型的基础——如果图文不能在向量空间里对齐，后面的 LLM 根本无法“看懂”图

配套文件位置：`docs/chapter2/code/clip_similarity_demo.py`（可复用）

## 七、对齐之后还差什么

CLIP 解决的是“相似不相似”的问题，它擅长判别式匹配，但不擅长自由生成。你不能仅靠 CLIP 直接完成一个复杂的开放式视觉问答系统。

因此，后续生成式多模态模型通常还会加入：

- 一个 Connector 或 Projector，把视觉特征映射进语言空间
- 一个 LLM，负责把这些特征组织成答案
- 更贴近助手场景的指令数据

## 八、常见对齐方式

### 1. 全局对齐

把整张图片压成一个全局向量，再和文本向量对齐。适合检索、粗粒度匹配，但细节能力有限。

### 2. Token 级对齐

保留 patch token 或 region token，让模型学习更细粒度的图文关系。这对 OCR、定位（Grounding）、复杂推理更有帮助。

### 3. 通过 Connector 对齐到 LLM

视觉编码器输出的特征，经过线性层、MLP（多层全连接网络）、Cross-Attention（让文本去"看"图像特征的注意力机制）或 Query 模块，转成 LLM 能吸收的输入表示。这是现代 VLM 的常见做法。

## 九、工程上如何判断“对齐没做好”

如果一个多模态模型出现下面这些现象，通常说明视觉信息没有真正进入推理链路：

- 对任何图片都回答模板化内容
- 回答主要依赖文字提示，而不是图像证据
- 图像换了，但回答几乎不变
- 图像中出现多个对象时，模型经常混淆主次
- 遇到细节问答时大面积幻觉

这时候不要急着只改 prompt，更应该回头检查：

- 输入分辨率和预处理是否合理
- 模型是否支持你当前的任务类型
- 服务端是否正确处理了图像输入
- 是否用了错误的 chat template 或 content schema

### Debug 清单（建议顺序）

建议按这个顺序排查，别一上来就猛改 prompt：

1. **输入链路**：图片是否真的传到服务端，路径/URL 是否有效。
2. **预处理链路**：分辨率、裁剪、压缩是否导致关键信息丢失。
3. **协议链路**：`messages.content` 结构是否符合当前模型接口要求。
4. **模型链路**：当前模型是否擅长你的任务（OCR、图表、UI 截图差异很大）。
5. **提示链路**：问题是否明确要求“基于图像证据回答”。

实际运行中，输入链路问题常常表现为 “FileNotFoundError: Image not found” 或 “No image found in message”；预处理链路问题则可能抛出 “Invalid image format” 或 “resolution exceeds limit” 的警告。这些日志能帮助你快速定位问题。

协议链路侧，OpenAI 兼容服务在 `messages` 缺图、字段名写错或 `image_url` 为空时，常见返回体里是 `400` / `422` 与 `message: image is required` 一类提示（具体文案因实现而异）。看到这类错误应优先对照第七章、第八章里的请求示例，而不是先怀疑“模型变笨了”。

## 十、理解视觉编码器时的三个视角

建议你以后看任何多模态论文或仓库时，都用这三个问题去拆：

1. 输入图像如何切分与编码？
2. 输出的是全局向量、局部 token，还是多尺度表示？
3. 这些视觉特征如何与语言模型连接？

只要这三点清楚，模型结构就基本不会看乱。

## 十一、实战衔接：自己观察分辨率和细节丢失

你现在就可以做一个简单实验，不需要训练任何模型：

1. 准备一张包含小字、按钮、图标的截图。
2. 保存原图，再手动缩小成一个低分辨率版本。
3. 分别把两张图喂给同一个多模态模型，问同样的问题。
4. 记录两次回答里哪些细节丢失了。

这个实验会非常直观地告诉你：视觉编码器和输入分辨率对结果影响有多大。

进一步，你还可以观察：

- 自然图像和 UI 截图在感知难度上的差异
- “描述图片”与“回答细节问题”对视觉 token 的要求差异
- 长图、密集表格、代码截图为什么更难

如果你愿意再往前一步，可以把实验结果整理成三列记录：

- 图像版本（原图/缩放/切块）
- 问题类型（描述/OCR/细节定位 Grounding）
- 错误类型（漏看/看错/推错）

这张小表后面做评测会非常有用，很多“感觉问题”都会变成可比较的数据。
