# 第四章 数据、训练与微调

> **理论篇**（第 1～4 章）：地图、视觉编码与对齐、生成架构、数据与微调。本章是理论篇收束，下一章起进入**实战篇**（评测与落地）。

## 一、为什么这章先聊数据

在纯文本 LLM 里，预训练数据已经足够复杂；到了多模态领域，数据复杂度会进一步上升，因为你不仅要处理文本本身，还要处理：

- 图像内容是否清晰
- 图文是否真正对应
- 标注是否稳定
- 指令是否有用
- 是否覆盖 OCR、图表、文档、自然图像等不同场景

所以在多模态里，决定上限的往往不是“用了哪个架构名词”，而是数据到底怎么配。

承接第三章：架构如果是“视觉编码器 + Connector + LLM”，那么预对齐阶段吃的是“视觉能否接上语言空间”的图文对，指令微调阶段吃的是“能否按用户意图完成任务”的指令样本。你本章配的数据类型和比例，基本就预定了模型上线后更像“会描述”还是更像“会办事”。

若样本里的图像分辨率、裁剪方式与 [第二章](../chapter2/第二章%20视觉编码器与跨模态对齐.md) 讨论的 patch 粒度脱节，后面在 [第五章](../chapter5/第五章%20评测体系与工程选型.md) 里看到的“小字漏读、表格错行”往往并不是评测脚本写错，而是训练分布里早就缺了那一档视觉细节。

## 二、常见数据类型

### 1. 图文描述数据

最基础，通常用于建立图像和自然语言之间的语义联系。

### 2. 图像问答数据

强化模型围绕问题提取证据、组织回答的能力。

### 3. OCR 与文档数据

用于提升读字、读版面、读表格、读公式的能力。对企业场景非常重要。更偏“票据合同、版面流水线、字段抽取”的打法，可与仓库 [Extra01 OCR 与文档理解专题](../Extra-Chapter/Extra01-OCR与文档理解专题.md) 穿插阅读，再回到本章把样本写进 JSONL。

### 4. Grounding / Referring 数据

让模型学会“定位”与“指代”，知道用户说的“左上角那个红色图标”到底是哪一块区域。

### 5. 推理数据

图表推理、视觉数学、多步比较、流程判断等，帮助模型从“看见”走向“推理”。

## 三、一个最小可用的数据样式

很多指令微调数据，本质上都可以抽象成下面这种结构：

```json
{
  "id": "sample-001",
  "image": "images/sample.png",
  "messages": [
    {
      "role": "user",
      "content": [
        { "type": "image", "image": "images/sample.png" },
        { "type": "text", "text": "请描述图中发生了什么。" }
      ]
    },
    {
      "role": "assistant",
      "content": [
        { "type": "text", "text": "图中一名骑行者正在穿过城市街道..." }
      ]
    }
  ]
}
```

注意顶层 `image` 字段用于快速索引和校验，`content` 中的 `image` 供训练框架读取——两者路径应保持一致。

这个格式的意义在于，它天然支持：

- 单图对话
- 多轮对话
- 多图输入
- 文本和图像混合内容

## 四、训练流程可以如何理解

多模态训练可以先粗分成三层：

### 1. 视觉与语言预对齐

目标是让视觉表示进入语言空间。常见做法是冻结主体，只训练中间模块。

### 2. 指令微调

目标是让模型学会按任务说话，而不是只做通用描述。

### 3. 偏好对齐或任务特化

当你想让模型更像一个产品，而不是一个通用 Demo，就会继续进行：

- 人类偏好对齐
- 领域数据继续微调
- 安全和格式约束强化

## 五、LoRA/QLoRA 在多模态里的位置

对于大多数学习者来说，整模型全量微调往往成本过高。更常见的是：

- 冻结视觉编码器
- 冻结大部分 LLM
- 只训 Projector
- 或者对 LLM 某些层做 LoRA

这种方式的优点很明显：

- 显存压力小
- 训练更快
- 更适合做场景适配

但要注意，LoRA 不是万能的。如果原模型在视觉侧本来就弱，或者训练数据分布差异很大，单靠 LoRA 也很难救回来。

### LoRA 关键参数怎么选

LoRA 的核心是往原模型的特定层里注入低秩矩阵，训练时只更新这些少量参数。三个最关键的设置：

| 参数 | 含义 | 常见取值 | 影响 |
|---|---|---|---|
| **rank (r)** | 低秩矩阵的秩 | 8, 16, 32, 64 | 越大表达能力越强，显存和训练时间也越高 |
| **alpha** | 缩放系数 | rank 或 2 × rank | 控制 LoRA 输出对原模型的影响强度，越大对原模型影响越强 |
| **target_modules** | 注入哪些原始层 | 因模型而异 | 决定哪些注意力/FFN 层参与微调 |

**VLM 微调的特殊考量**：

1. **Vision Tower 冻不冻？**
   - 默认做法：冻结视觉编码器（它已经“会看”了）
   - 例外：如果你的场景视觉差异极大（如医学影像、卫星图），可尝试给 vision tower 也上 LoRA，但显存会显著增加

2. **Projector / Connector 冻不冻？**
   - 通常**不冻**，因为场景适配往往需要视觉-语言对齐的微调
   - 如果只做纯文本风格迁移（如让模型说话更像客服），可冻结 projector 只训 LLM 侧

3. **Target modules 怎么查？**
   - Qwen2.5-VL: `["q_proj", "k_proj", "v_proj", "o_proj", "gate_proj", "up_proj", "down_proj"]`
   - LLaVA: `["q_proj", "v_proj"]`（早期版本只调这两个就够了）
   - 通用原则：先调 attention 的 q/v_proj，效果不够再加 FFN 层

4. **QLoRA 是什么？**
   - 把基座模型先量化为 4-bit（NF4），再在 4-bit 模型上加 LoRA
   - 效果：7B 模型可在 16GB 显存上微调，24GB 可微调 13B（具体显存与序列长度、batch size、gradient checkpointing 等相关，此处为典型参考值）
   - 代价：推理时需合并权重或保持量化状态，轻微精度损失

配套代码见 `docs/chapter4/code/lora_finetune_minimal.py`，目标是在 Colab T4 / 24GB 卡上对 Qwen2.5-VL-3B 做 100 条数据的 caption 微调，建立“数据 → 训练 → 验证”的闭环手感。

## 六、哪些模块该冻结，哪些模块该训练

一个很实用的判断方式是先问自己：**你到底想改什么能力？**

| 目标 | 建议优先训练的部分 |
| --- | --- |
| 只是把视觉接进来 | Projector / Connector |
| 提升回答风格与任务遵循 | LLM 部分层 + 指令数据 |
| 提升 OCR、版面、细节理解 | 更强视觉侧、分辨率策略、相关数据 |
| 提升领域知识 | 领域数据 + 语言侧适配 |

如果问题本质上来自视觉感知不足，你却只去改语言模型，通常收益有限。

在动显卡之前，可以用三个问题自检，避免“为训而训”：

1. **失败主因更像视觉漏看、还是更像指令/格式没对齐？** 前者优先补数据与分辨率策略，后者优先统一 chat template 与答案风格。
2. **基座模型在目标场景上是否已经“勉强可用”？** 若完全不可用，先换模型或先缩小场景，再谈 LoRA。
3. **是否已有一批 JSONL + 离线 eval 能对齐迭代？** 没有验收集时扩数据容易变成扩噪声。

## 七、偏好对齐：减少幻觉的主流方向（2024-2026）

SFT 让模型“会回答”，但回答可能包含幻觉（没看图却瞎编）。2024 年后，减少 VLM 幻觉的主流方法从“加更多 SFT 数据”转向“偏好对齐”——让模型学会“什么样的回答更好”。

### 常见方法速览

| 方法 | 核心思想 | 数据需求 | 适合场景 |
|---|---|---|---|
| **DPO** | 直接比较“好回答”和“差回答”的似然，无需奖励模型 | 成对偏好数据 | 快速上线，资源有限 |
| **KTO** | 只需要知道“这条回答好/不好”的二元信号 | 二元反馈数据 | 比 DPO 数据收集更简单 |
| **RLHF-V** | 用人类反馈做强化学习，专门优化视觉忠实度 | 人类标注的偏好对 | 幻觉严重的场景 |
| **RLAIF-V** | 用 AI（如 GPT-4V）替代人类提供偏好判断 | AI 生成的偏好对 | 降低标注成本 |
| **VLFeedback** | 大规模 VLM 偏好数据集，覆盖多种视觉任务 | 现成数据集 | 研究/通用能力提升 |
| **POVID** | 针对“物体存在性”的偏好优化，减少“图中没有却说有” | 物体存在性标注 | 细粒度幻觉抑制 |

### 对教程读者的务实建议

1. **先做好 SFT**：偏好对齐的前提是模型已经能“基本答对”，不要跳步
2. **DPO 是最小可行路径**：如果你有 500~2000 条“好回答 vs 差回答”的配对，DPO 是性价比最高的选择
3. **幻觉评测优先用 POPE / HallusionBench**：第五章提到的这两个 benchmark 专门测幻觉，对齐前后务必跑一遍对比
4. **工具链**：TRL 库的 `DPOTrainer`、LLaMA-Factory 的 DPO 模式、OpenRLHF 都支持多模态 DPO

> 配套代码：本仓库暂不提供完整 DPO 脚本（需要成对偏好数据，数据构建比 SFT 复杂），但 `lora_finetune_minimal.py` 的训练框架可直接对接 TRL 的 DPOTrainer。参考方向：TRL 官方文档 + 你的 SFT 数据生成“拒绝采样”版本的负例。

## 八、多模态训练的常见坑

### 1. 图文错配

图像和文本表面上看似对应，但内容不一致。这会直接污染对齐效果。

### 2. 过度依赖描述型数据

如果训练集几乎都是“请描述图片”，那模型很可能学会了看图写说明文，却不擅长复杂问答。

### 3. 忽略 OCR 与文档样本

自然图像训练得很好，不代表能读发票、PPT、网页截图。

### 4. Prompt 模板前后不一致

训练和推理使用了不同的消息模板，容易导致回答风格和能力不稳定。

### 5. 只盯着 loss，不看抽样结果

多模态训练里，人工抽样看结果很重要。loss 下降不等于模型真的“看懂了”。抽样时最好对照第五章的维度想一想：这条样本是在测感知、OCR、推理还是幻觉？避免“训练集全是描述句、上线却要问细节”这种隐性错配。

**工程速记**：有人 SFT 后 loss 曲线很漂亮，同一批业务截图在评测集上却仍只会泛泛描述；复盘 JSONL 才发现几乎全是“请描述这张图”，缺少带证据指向的问答对。这类配方问题不会写进单个 loss 数字里，却会原样暴露在第五章按 `tags` 拆开统计的失败率上。

## 九、怎么做一个场景化小微调方案

本节侧重**策略决策**（选什么任务、用什么数据、冻什么模块），与[第十一节“实战：做一个最小多模态 SFT 数据集”](#十一实战做一个最小多模态-sft-数据集)的动手执行互补。

如果你是第一次做场景微调，我建议按这个顺序来：

1. 先明确任务边界，例如电商图文、文档抽取、截图问答。
2. 收集 200 到 2000 条高质量样本，不追求一开始就大规模。
3. 保持统一消息格式和答案风格。
4. 先做 Connector 或 LoRA 微调，快速看收益。
5. 抽样人工评估，再决定是否继续扩充数据。

这比一上来追求“大而全”靠谱得多。

## 十、训练之外，数据治理别省

教程项目经常忽略一个问题：真正能用于反复迭代的数据，必须可以治理。

建议至少记录：

- 数据来源
- 标注规则
- 清洗标准
- 任务标签
- 图像分辨率和格式
- 是否涉及隐私与敏感信息

这些信息平时看着啰嗦，但等你要复盘、扩数据、协作时会非常救命。

## 十一、实战：做一个最小多模态 SFT 数据集

从这部分开始，最好真的做一个“小而完整”的数据实验，而不只是停在概念层面。

仓库里配套文件包括：

- `docs/chapter4/code/sample_multimodal_sft.jsonl`
- `docs/chapter4/code/prepare_dataset_manifest.py`
- `docs/chapter4/code/create_placeholder_images.py`（首次校验前生成 JSONL 里引用的占位图，避免全是 `image_exists: false`）

直接这样跑就行：

```bash
cd docs/chapter4/code
python create_placeholder_images.py
python prepare_dataset_manifest.py
# 或在仓库根目录：
# python docs/chapter4/code/prepare_dataset_manifest.py --dataset docs/chapter4/code/sample_multimodal_sft.jsonl
```

实操时按这个顺序操作：

1. 先阅读 `sample_multimodal_sft.jsonl`，理解一条图文指令数据应该长什么样。
2. 运行 `create_placeholder_images.py` 得到最小占位图，或把路径改成你自己的真实样本。
3. 用 `prepare_dataset_manifest.py` 对数据做一次基础检查。
4. 对照打印结果，确认图片是否存在、消息轮次是否完整、文本是否为空。

### 一个最小实验思路

如果你暂时不想训练大模型，也可以先做下面这个闭环：

1. 自己做 20 条图文问答数据。
2. 按统一格式保存成 JSONL。
3. 用脚本校验格式。
4. 抽样人工检查答案风格是否一致。
5. 再决定是否继续扩到 100 条、500 条。

这一步非常适合把“学习数据配方”变成“真正会做数据”。

## 十二、章末练习

**动机**：数据配方是多模态效果的“第一因”；练完你应能解释“同一套架构、不同 JSONL 为什么会训出两种产品形态”，并敢动手写最小可训样本。

### 必做（约 15 分钟）

1. 列出一个你想做的场景，并给出至少 4 类你需要的数据类型。
2. 解释为什么过度依赖“描述型数据”会限制模型的问答能力。

### 进阶（约 30 分钟）

1. 如果资源有限，你会优先训练 Projector、LoRA 还是全量参数？说明理由。
2. 按本章的数据格式，手写 3 条你自己的多模态 SFT 样本，并用 `prepare_dataset_manifest.py` 能跑通的路径保存。

### 挑战（1～2 小时）

1. 从你手写的样本里抽 5 条，改写为 [第五章](../chapter5/第五章%20评测体系与工程选型.md) 评测 JSONL 字段（`image` / `question` / `reference` / `tags`），跑一遍 `eval_vlm_dataset.py`（可不训练，只看基座模型在自家数据上的失败模式）。

## 十三、带走这三件事

- 多模态模型的训练效果很大程度上由数据配方决定。
- 预对齐解决“接得上”，指令微调解决“用得顺手”，任务特化解决“能落地”。
- 资源有限时，优先做高质量小规模实验，比盲目扩大规模更有效。

**第五章**把同一批场景收成可重复的离线验收：你本章整理好的图片与问题，可直接改写到 `docs/chapter5/code` 下的评测 JSONL。训练集与评测集共用同一套消息习惯，迭代才不会自说自话。
